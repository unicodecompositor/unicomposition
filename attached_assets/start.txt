Разворачивание проекта связка project->dev->github->server(test_prod)

Редактировать .env проекта
GATE_USER=
GATE_NAME_PROJECT=
GATE_ADMIN_USER=
GATE_SERVER=
GET_SSH=

Редактируем GitHub переменные
| GitHub	| Источник данных	| Команда для копирования |
| HOST | .env → GATE_SERVER | grep GATE_SERVER .env |
| USERNAME |	.env → GATE_USER |grep GATE_USER .env |
| SSH_KEY | Файл id_rsa	cat | ./id_rsa (копируй от BEGIN до END) |
| PORT | .env → GATE_SSH | grep GATE_SSH .env |

Теперь твой флоу подготовки (один раз):

    bun run setup:user — создает юзера, дает ему права на докер и готовит папку /var/www/unicomp.
    bun run setup:keys — готовит ключи.
    bun run init:project — прописывает known_hosts и права на rm (через наш прошлый скрипт).
    bun run clean:secrets — чистит секреты.
    

1. Подготовь .env.example
В архив или публичный репо нельзя класть реальный .env с твоими IP и ключами. Вместо него положи файл с описанием:

# Имя твоего проекта (будет названием папки в /var/www/)
GATE_NAME_PROJECT=unicomp-template
# Твой логин на сервере с правами sudo
GATE_ADMIN_USER=admin
# IP или домен сервера
GATE_SERVER=0.0.0.0
# Порт SSH (по умолчанию 22)
GATE_SSH=22
# Временный файл-флаг для открытия доступа
GATE_TOKEN_FILE=/tmp/ssh_access_open

2. Добавь инструкцию в README.md
В разделе «Для обучения» напиши, что этот проект решает проблему безопасного деплоя без открытых портов.

* Объясни, что init:project — это «магическая кнопка», которая настраивает сервер с нуля.
* Укажи, что используется AuthorizedKeysCommand — это киллер-фича твоего сборщика.

3. Скрипт-автомат в package.json
У тебя в package.json уже всё проименовано и логично связано. Человек просто:

   1. Клонирует репо.
   2. Копирует .env.example -> .env.
   3. Запускает bun run init:project.
   4. И у него всё работает.

Что в итоге получит ученик:
Он получит не просто сайт, а готовую схему: Bun + Vite + Docker + GitHub Actions + SSH Hardening. Это уровень крепкого мидла в DevOps.
Совет: Перед распространениме убедись, что удалил сами файлы ключей (id_rsa), если уже их генерировал, чтобы не подарить свой доступ всему интернету.

Для распространения проекта и очистки от ключей а так же файла .enn предусмотрена команнда clean:secrets

⚠️ **ВНИМАНИЕ: БЕЗОПАСНОСТЬ ПРЕЖДЕ ВСЕГО**

Перед тем как публиковать свой форк или делиться архивом:
1. Убедитесь, что файлы `id_rsa`, `id_rsa.pub` и `id_rsa_deploy.pub` добавлены в `.gitignore`.
2. НИКОГДА не коммитьте файл `.env` с реальными IP и именами пользователей.
3. Если вы уже сгенерировали ключи, удалите их перед распространением проекта: `rm id_rsa*`.
4. Новые пользователи должны будут сами запустить `bun run setup:keys` после клонирования.
